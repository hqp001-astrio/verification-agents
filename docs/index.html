How it works

Agent Architecture

The orchestrator is a GPT-4o agent running in a tool-use loop — not a fixed pipeline. It decides what to do at each step, can ask clarifying questions, and ends only when it calls submit_report.

🧠
Orchestrator Agent
GPT-4o in a tool-use loop · up to 20 turns
▼

The orchestrator receives the raw diff and a system prompt that instructs it to follow the verification workflow. It calls tools in sequence, interprets results, and writes the final prose summary inside submit_report. It cannot finish without calling that tool.

Model

gpt-4o (default)

Max turns

20 (configurable)

Input

Unified diff + optional intent

Output

VerificationReport (JSON)

🌳
parse_diff
tree-sitter AST · call graph · property discovery
▼

Parses the diff headers to find changed files and hunk ranges. Uses tree-sitter to build an AST for each file, then walks it to extract changed functions. Scans each function for verifiable patterns: array subscripts, attribute accesses, arithmetic, and loops.

Input

Raw unified diff string

Output

CodeAnalysis (units + call graph + properties)

Languages

Python, JavaScript/TypeScript

Parser

tree-sitter (zero subprocess)

💬
ask_user
presents options · waits for selection · extensible handler
▼

The agent never verifies properties without user confirmation. It presents the discovered properties and calls the injected clarification_handler. This is the seam for the GitHub bot layer — the bot can post a PR comment and wait for a reply without changing any agent code.

Input

Question + list of VerifiableProperty

Output

UserSelection (selected IDs + notes)

Handler

Injected by caller (CLI, bot, test)

Bot hook

Post PR comment, await reply

⚗️
formalize
inner LLM call · generates Z3 Python expressions
▼

Makes a focused GPT-4o call per selected property, asking it to produce a Python snippet that defines a z3.BoolRef named constraint. The snippet is executed in a sandboxed scope. The resulting live Z3 objects are stored in a module-level registry keyed by property ID.

Input

UserSelection + CodeAnalysis

Output

list[Z3Constraint] + in-memory z3.BoolRef store

Execution

exec() in restricted scope

Fallback

Skips constraint if z3.ExprRef not produced

🔬
z3_solve
negation-based checking · counterexample extraction
▼

For each constraint P, asserts Not(P) and calls solver.check(). If SAT, the property can be violated — a bug. Z3 provides a concrete model (counterexample) showing which variable values trigger it. UNSAT means the property always holds.

SAT result

Bug found — extract counterexample from model

UNSAT result

Property holds for all inputs

UNKNOWN

Timeout — reported as inconclusive

Timeout

30s per constraint (configurable)

📋
submit_report
severity mapping · assembles VerificationReport · ends loop
▼

Maps each SAT result to a Bug with a severity level based on the property kind. The orchestrator writes the prose summary (including counterexample values) directly into the tool call arguments. Receiving this result terminates the agent loop.

Input

SolverResult list + prose summary

Output

VerificationReport (JSON-serializable)

Terminates

Agent loop ends on receipt

Summary

Written by orchestrator, not this tool

Classification

Severity Levels

Bugs are classified by property kind. The orchestrator sorts them critical-first in the report.

Severity map

Tool API

Each tool is a Python function exposed to the orchestrator as an OpenAI function-calling tool. Click a tab to explore.

parse_diff is always the first tool the agent calls. It turns a raw unified diff into a structured CodeAnalysis with extracted functions, a call graph, and a menu of verifiable properties.
Python
from verification_agents.tools.parse_diff import run

diff = open("feature.diff").read()
analysis = run(diff)

print(f"{len(analysis.units)} functions changed")
for p in analysis.properties:
    print(f"  [{p.kind}] {p.unit_name}: {p.description}")

ask_user is the seam between the agent and the caller. Inject a clarification_handler — the bot layer posts a GitHub PR comment; the CLI prompts the terminal; tests auto-select all.
Python — CLI handler
from verification_agents.models import UserSelection, VerifiableProperty

def terminal_handler(question: str, options: list[VerifiableProperty]) -> UserSelection:
    print(question)
    for i, opt in enumerate(options):
        print(f"  {i+1}. [{opt.kind}] {opt.unit_name}: {opt.description}")
    raw = input("Select (comma-separated) or Enter for all: ").strip()
    ids = [o.id for o in options] if not raw else [
        options[int(x)-1].id for x in raw.split(",")
    ]
    return UserSelection(selected_ids=ids)
Python — test / auto-select handler
def auto_select(question: str, options: list[VerifiableProperty]) -> UserSelection:
    return UserSelection(selected_ids=[o.id for o in options])
formalize makes a focused GPT-4o call per selected property to generate a Z3 Python expression. The resulting z3.BoolRef objects live in a module-level store and are retrieved by z3_solve via property ID.
Example Z3 snippet produced by GPT-4o
# For: array_bounds — "verify index i is within bounds of items"
import z3
i = z3.Int("i")
n = z3.Int("n")   # n = len(items)
constraint = z3.And(i >= 0, i < n)
For: null_dereference — user.membership access
import z3
user_is_not_none = z3.Bool("user_is_not_none")
constraint = user_is_not_none  # must be True for safe access
z3_solve uses negation-based checking: it asserts Not(P) and asks Z3 to find an input that satisfies the negation. SAT = bug found with concrete counterexample. UNSAT = property always holds.
Python — what the solver does internally
import z3

# Property P: 0 <= index < len(array)
index = z3.Int("index")
length = z3.Int("length")
P = z3.And(index >= 0, index < length)

solver = z3.Solver()
solver.set("timeout", 30_000)
solver.add(z3.Not(P))   # negate: can P be violated?

result = solver.check()
if result == z3.sat:
    model = solver.model()
    # e.g. {"index": "5", "length": "3"}  ← concrete bug witness
    counterexample = {str(d): str(model[d]) for d in model.decls()}

Setup

Quick Start

Prerequisites: Python 3.11+, uv, an OpenAI API key.

1

Clone and install

bash
git clone <repo-url> verification-agents
cd verification-agents
uv sync
2

Set your API key

bash
echo "OPENAI_API_KEY=sk-..." > .env
3

Run against the sample repo

bash
# uses ../sample-buggy-app by default
uv run python run.py

# or point at any repo + branches
uv run python run.py /path/to/repo main feature/my-branch
4

Use as a library

Python
from verification_agents.orchestrator import OrchestratorAgent
from verification_agents.models import UserSelection, VerifiableProperty

def my_handler(question, options: list[VerifiableProperty]) -> UserSelection:
    # auto-select all for CI / GitHub bot
    return UserSelection(selected_ids=[o.id for o in options])

agent = OrchestratorAgent(
    api_key="sk-...",
    clarification_handler=my_handler,
    model="gpt-4o",         # default
    z3_timeout_ms=30_000,   # default
)

report = agent.run(diff_text, user_intent="check for null dereferences")

print(report.model_dump_json(indent=2))
